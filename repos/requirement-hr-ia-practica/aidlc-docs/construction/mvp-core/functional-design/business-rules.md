# Business Rules — Unit 1: MVP Core

## Overview

Business rules are pure functions defined in domain rule classes. They encode invariants, validations, and decision logic that must hold regardless of infrastructure.

---

## BR-CONV: Conversation Rules

**Location**: `src/domain/conversation/ConversationRules.ts`

### BR-CONV-01: Phase Transition Guard

A conversation can only advance to the next phase if the current phase is complete.

```
onboarding   → consent      : always allowed (onboarding is informational)
consent      → verification : ONLY IF consent.granted = true
consent      → abandoned    : IF consent.granted = false
verification → screening    : ONLY IF all mandatory requirements passed
verification → abandoned    : IF any mandatory requirement failed
screening    → closing      : ONLY IF all competencies covered (competenciesCovered.length = rubric.competencies.length)
closing      → [end]        : conversation.state = 'completed'
```

### BR-CONV-02: Consent Gate

No evaluative question (verification or screening) may be asked before consent is recorded as `granted = true`.

### BR-CONV-03: Follow-Up Rule (A2)

After each competency question, exactly one follow-up (repregunta) is always asked. The follow-up is generated by the LLM based on the candidate's response and the competency criteria.

```
For each competency:
  1. Ask main competency question
  2. Receive candidate response
  3. Ask exactly one follow-up question
  4. Receive follow-up response
  5. Mark competency as covered
  6. Move to next competency
```

### BR-CONV-04: Competency Order (A3)

At session start (transition to `screening` phase), the competency IDs from the rubric are shuffled randomly and stored in `sessionState.competencyOrder`. This order is fixed for the duration of the session.

```
competencyOrder = shuffle(rubric.competencies.map(c => c.id))
currentCompetencyIndex = 0
```

### BR-CONV-05: Screening Completion Condition

Screening is complete when all competencies in `competencyOrder` have been covered (main question + follow-up answered).

```
isScreeningComplete = sessionState.competenciesCovered.length === rubric.competencies.length
```

### BR-CONV-06: Escalation Level Rules (A5)

```
Level 1 (info_not_available):
  - Triggered when: candidate asks a factual question not answerable from knowledge base
  - Action: log escalation, respond with standard message, add topic to unansweredTopics
  - Standard response: "No tengo esa información. El equipo de reclutamiento te dará los detalles en la siguiente etapa."

Level 2 (repeated_question):
  - Triggered when: same topic appears 2+ times in unansweredTopics
  - Action: log escalation, suggest human follow-up, increment escalationLevel to 2
  - Response: "Entiendo que necesitas más información sobre este tema. Un reclutador podrá ayudarte en la siguiente etapa."

Level 3 (explicit_human_request):
  - Triggered when: candidate explicitly requests to speak with a human
  - Action: create EscalationRequest, notify dashboard, preserve session context
  - Response: "Entendido. He notificado al equipo de reclutamiento. Tu progreso está guardado."
  - Session continues (not terminated) — candidate can still complete screening
```

**Topic matching for L2**: Two questions are considered the same topic if the LLM classifies them as semantically equivalent (single classification call).

### BR-CONV-07: Session State Persistence (A4)

`SessionState` is written to DynamoDB after every message exchange (both agent and candidate messages). This ensures no progress is lost on server restart.

### BR-CONV-08: Inactivity Rules

```
5 minutes inactive  → send "Tómate tu tiempo" message (no state change)
24 hours inactive   → send reminder message (no state change)
48 hours inactive   → send final reminder (no state change)
72 hours inactive   → state = 'abandoned', candidate.state = 'abandoned'
```

Note: Re-engagement timers (24h/48h/72h) are deferred to Unit 4. MVP only implements the 5-minute in-session message.

### BR-CONV-09: AI Disclosure Rule

If the candidate asks "¿Eres IA?" or any semantically equivalent question at any point, the agent must confirm it is an AI. This check runs on every candidate message before routing to the normal flow.

## BR-EVAL: Evaluation Rules

**Location**: `src/domain/evaluation/EvaluationRules.ts`

### BR-EVAL-01: Batch Evaluation Timing (B1)

Evaluation runs once, after the entire screening is complete (all competencies covered + follow-ups answered). There is no partial/real-time scoring during the conversation.

```
Trigger: CompleteScreeningUseCase called
  → GenerateSummaryUseCase
    → Single LLM call per competency (score + evidence extraction)
    → Calculate global score
    → Determine recommendation
    → Save ExecutiveSummary
```

### BR-EVAL-02: Score + Evidence in Single LLM Call (B2)

Each competency is evaluated with a single LLM call that returns both the score (1-5) and the verbatim evidence quote.

```
Input to LLM:
  - Competency name and criteria (levels 1-5)
  - All candidate messages related to this competency (main + follow-up)
  - Full conversation context

Output from LLM (structured JSON):
  {
    "score": 1-5,
    "evidence": {
      "quote": "<verbatim candidate text>",
      "messageIndex": <index in messages array>,
      "relevance": "<why this quote supports the score>"
    }
  }
```

### BR-EVAL-03: Evidence Validation

Every CompetencyScore must have at least one Evidence item with a non-empty quote. If the LLM returns no quote, the system must retry the evaluation call once. If still no quote, use the full candidate response for that competency as the quote.

### BR-EVAL-04: Global Score Calculation (B3)

```
globalScore = sum(score_i × weight_i) / sum(weight_i)
```

Where `weight_i` is the competency weight from the rubric (all weights sum to 1.0, so denominator = 1.0). Result is rounded to 1 decimal place. Range: 1.0 – 5.0.

### BR-EVAL-05: Recommendation Thresholds (B4)

```
globalScore >= 4.0  → 'highly_recommended'
globalScore >= 2.5  → 'recommended'
globalScore <  2.5  → 'not_recommended'
```

### BR-EVAL-06: Key Signals Generation

The executive summary includes 3-5 key signals (notable observations). These are generated by the LLM as part of the summary generation call, based on the full transcript and competency scores.

### BR-EVAL-07: Rubric Weight Invariant

The sum of all competency weights in a rubric must equal 1.0 (±0.001 tolerance for floating point). This is validated at rubric creation time.

## BR-CAMP: Campaign Rules

**Location**: `src/domain/campaign/CampaignRules.ts`

### BR-CAMP-01: Mandatory Fields (E3)

A campaign cannot be activated without:
- `name` (non-empty string)
- `roleDescription` (non-empty string)
- `rubricId` (valid rubric reference)

`knowledgeBaseContent` and `basicRequirements` are optional.

### BR-CAMP-02: Telegram Link Generation (C2)

```
telegramLink = `https://t.me/${BOT_USERNAME}?start=${campaignId}`
```

Generated at campaign creation. `BOT_USERNAME` is read from environment configuration. The `campaignId` in the `start` parameter is used by the webhook handler to route the conversation to the correct campaign.

### BR-CAMP-03: Status Transitions

```
draft    → active    : allowed if mandatory fields present
active   → inactive  : always allowed
inactive → active    : always allowed
active   → archived  : NOT allowed (must deactivate first)
inactive → archived  : always allowed
archived → *         : NOT allowed (terminal state)
```

### BR-CAMP-04: Knowledge Base Injection (C1)

When a campaign has `knowledgeBaseContent`, the full text is injected into the system prompt at the start of every conversation for that campaign. The content is loaded once when `StartScreeningUseCase` runs and stored in `sessionState.knowledgeBaseLoaded = true`.

## BR-CAND: Candidate Rules

**Location**: `src/domain/candidate/CandidateRules.ts`

### BR-CAND-01: New Record Per Attempt (D1)

Each time a Telegram user starts a screening for a campaign, a new Candidate record is created. Previous records for the same `telegramUserId` + `campaignId` combination are not modified — they remain as historical data.

### BR-CAND-02: Name Collection (D2)

The bot asks for the candidate's name as the first question during onboarding, before any other interaction. The name is stored in `candidate.name`.

```
Onboarding sequence:
  1. AI disclosure message
  2. "¿Cuál es tu nombre?" → store response as candidate.name
  3. Explain process and rights
  4. Proceed to consent
```

### BR-CAND-03: State Transition Guard

```typescript
const VALID_TRANSITIONS: Record<CandidateState, CandidateState[]> = {
  initiated:       ['in_screening', 'abandoned'],
  in_screening:    ['completed', 'abandoned'],
  completed:       ['pending_review'],
  pending_review:  ['approved', 'rejected'],
  approved:        [],   // terminal
  rejected:        [],   // terminal
  abandoned:       [],   // terminal
};
```

### BR-CAND-04: Disagreement Capture (E4)

When a recruiter submits a review decision that differs from the AI recommendation, the `disagreementReason` field becomes required. The UI enforces this with a required text field that appears automatically when the decision differs.

```
IF reviewDecision.decision = 'approved' AND evaluation.recommendation = 'not_recommended'
  OR reviewDecision.decision = 'rejected' AND evaluation.recommendation IN ('recommended', 'highly_recommended')
THEN disagreesWithAI = true AND disagreementReason MUST be non-empty
```

## BR-COMP: Compliance Rules

**Location**: `src/domain/compliance/ComplianceRules.ts`

### BR-COMP-01: Consent Immutability

A ConsentRecord is written once and never updated. If a candidate starts a new session (new Candidate record), a new ConsentRecord is created.

### BR-COMP-02: Audit Event Immutability

AuditEvents are append-only. No update or delete operations are permitted on the audit_events table. This is enforced at the repository level (no `update` or `delete` methods on `AuditEventRepository`).

### BR-COMP-03: Mandatory Audit Events

The following events MUST be logged regardless of outcome:
- `consent_granted` / `consent_denied` — on every consent interaction
- `screening_started` — when conversation enters `screening` phase
- `screening_completed` / `screening_abandoned` — on conversation end
- `requirements_failed` — when candidate fails verification
- `evaluation_generated` — when ExecutiveSummary is saved
- `candidate_approved` / `candidate_rejected` — on every HITL decision
- `escalation_l1` / `escalation_l2` / `escalation_l3` — on every escalation

## BR-DASH: Dashboard Rules

### BR-DASH-01: Review Queue Default Sort (E1)

The HITL review queue defaults to all candidates with `state = 'pending_review'`, sorted by `createdAt` ascending (oldest first — FIFO processing order).

### BR-DASH-02: Tenant Scoping

Every dashboard query is automatically scoped to the authenticated recruiter's `tenantId` (extracted from Cognito JWT). No cross-tenant data is ever returned.

### BR-DASH-03: Transcript Display (E2)

The candidate detail view shows the full scrollable transcript below the executive summary. All messages are visible without pagination. Evidence quotes in the summary are highlighted/linked to their position in the transcript.

### BR-DASH-04: Data Freshness

Dashboard data is not real-time. Recruiters must manually refresh the page to see new entries. No WebSocket or polling is implemented in MVP.
